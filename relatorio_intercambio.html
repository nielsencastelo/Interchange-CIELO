Objetivo Central: Transformar documentos técnicos das Bandeiras (PDFs, HTMLs, CSVs) em dados estruturados e consultáveis, habilitando análises comparativas, simulações de custo e suporte à decisão estratégica.

2. Contexto e Motivação

O ecossistema de pagamentos brasileiro é um dos mais complexos e dinâmicos do mundo. Com mais de 6 Bandeiras ativas, múltiplos emissores e credenciadores, e regulamentação do Banco Central do Brasil (BCB), as taxas de intercâmbio representam um componente crítico da equação econômica de qualquer estabelecimento ou instituição financeira.

Os manuais de intercâmbio publicados pelas Bandeiras têm centenas de páginas, atualizações trimestrais e estruturas não padronizadas. Extrair e comparar essas regras manualmente é inviável em escala.

Desafio técnico principal: Os documentos são heterogêneos — PDFs nativos, PDFs escaneados, HTMLs, planilhas — e usam terminologia própria de cada Bandeira, dificultando a extração automatizada sem normalização semântica.

3. Análise Exploratória dos Documentos

3.1 Estrutura Tarifária das Bandeiras

Após análise dos documentos públicos disponíveis, identificou-se que todas as Bandeiras organizam suas regras de intercâmbio em camadas aditivas:

  1. Taxa Base: determinada pelo produto do cartão (Classic, Platinum, Black, etc.)
  2. Ajuste por Segmento: incremento ou redução baseado no MCC do estabelecimento
  3. Ajuste por Canal: contactless, CNP (e-commerce), ATM
  4. Ajuste por Parcelamento: adicional por número de parcelas
  5. Cap/Floor: limites absolutos por transação (regulatórios ou comerciais)

4.1 Detalhamento das Etapas

  • Ingestão: Suporte a PDF nativo (pdfplumber), PDF escaneado (OCR via pytesseract), HTML (httpx + parser) e CSV estruturado.
  • Chunking: Divisão por blocos de ~800 chars, respeitando limites de linha para não fragmentar tabelas.
  • Regex: 15+ padrões cobrindo percentuais, valores BRL/USD, bandas de parcelamento, caps e termos de canal.
  • LLM: Suporte a Anthropic Claude, OpenAI GPT, Google Gemini e Ollama (local). A LLM é invocada apenas para chunks ambíguos, reduzindo custo.
  • Validação: Score de confiança 0–1 baseado em 10 critérios (faixa de taxa, contexto, campos preenchidos).
  • Persistência: SQLAlchemy com suporte a SQLite (dev) e PostgreSQL (produção), com versionamento por tag.

7. Desafios Técnicos Identificados

  • PDFs escaneados: Cerca de 40% dos documentos históricos são imagens digitalizadas, exigindo OCR antes da extração de texto. Solução: pytesseract + pré-processamento de imagem.
  • Tabelas aninhadas: Estruturas de tabela com rowspan/colspan perdem formatação na extração de texto. Solução: pdfplumber com configurações de tolerância + LLM para reconstrução.
  • Terminologia não padronizada: Cada Bandeira usa nomes diferentes para o mesmo conceito (ex: "World Legend" vs "Infinite"). Solução: tabela de mapeamento canônico em normalizer.py.
  • Atualizações frequentes: Manuais são atualizados trimestralmente. Solução: versionamento por version_tag + DAG Airflow para automação.
  • Ambiguidade semântica: Regras com condicionais implícitas ("exceto para segmentos X e Y"). Solução: LLM com prompt estruturado + revisão humana para baixo score.
  • Multilíngue: Documentos em português (BR), inglês e espanhol. Solução: normalização de texto + suporte a regex bilíngue.

8. Recomendações Técnicas para o Agente de IA

8.1 Curto Prazo (MVP)

  • Usar o pipeline atual (regex + Claude/GPT) para processar os manuais mais recentes.
  • Implementar fila de revisão humana para regras com confidence_score < 0,50.
  • Integrar OCR (pytesseract) para documentos escaneados.

8.2 Médio Prazo

  • Treinar modelo de NER (Named Entity Recognition) customizado para entidades de intercâmbio (taxa, produto, segmento, canal).
  • Implementar Document AI (Google Document AI ou AWS Textract) para extração de tabelas complexas.
  • Criar base de knowledge graph ligando Bandeira → Produto → Regra → Segmento.

8.3 Longo Prazo (Nível Doctoral)

  • Fine-tuning de LLM especializado em documentos financeiros brasileiros (usando os manuais coletados como corpus).
  • Implementar Retrieval-Augmented Generation (RAG) sobre os documentos, permitindo perguntas em linguagem natural como "Qual a taxa para Platinum em supermercado parcelado 6x na Visa?"
  • Sistema de detecção automática de mudanças nas regras entre versões de documentos (diff semântico).
  • Integração com dados públicos do BCB (PAAR API) para validação automática das taxas extraídas.

9. Conclusão

O pipeline Interchange AI demonstra que é possível automatizar com alta precisão a extração de regras tarifárias de documentos técnicos das Bandeiras, combinando abordagens determinísticas (regex) e probabilísticas (LLM).

A arquitetura modular permite evolução incremental: iniciar com extração por regex, adicionar LLM para casos ambíguos, e eventualmente treinar modelos especializados.

Contribuição para o estado da arte: A combinação de extração determinística + normalização LLM + score de confiança + revisão humana representa uma abordagem Human-in-the-Loop (HITL) adequada para domínios regulatórios onde a precisão é crítica.
